# F1 — E1.1 demand probe results

Date: 2026-04-30. Branch: `exp/v0.2-validation`.

## Question

Which Humble camera drivers in active use today emit
`sensor_msgs/Image` with `encoding == "nv12"`? Outcome decides
whether a `prism::FormatBridgeNode` has any users on stock Humble
or only on speculative future hosts.

## Method

Read-only source survey of six drivers, plus the cross-cutting
conversion path. For each, located either the encoding-string
table or the GstCaps mapping, and recorded what the driver can
actually publish.

## Results

| Driver | NV12 today | Mechanism | Gap |
| --- | --- | --- | --- |
| `ros2_v4l2_camera` (boldhearts / tier4) | **No** | Only `YUYV`, `UYVY`, `GREY` documented | Source patch required to add NV12 V4L2 path |
| `gscam2` (clydemcqueen, upstream) | **No** | Caps mapping handles `RGB8`, `MONO8/16`, `YUV422_YUY2`, Bayer, JPEG only | Source patch required to add NV12 caps branch |
| `gscam2` (TI Robotics SDK fork) | **Yes — default** | NV12 is the default output for integration with `ti_vision_cnn` | TI Jacinto SoC family only (J7, AM68A, AM62A) |
| `usb_cam` (ros-drivers) | **No** | Supports `yu12` (planar YUV420, separate U/V planes) — distinct from NV12 (semi-planar, interleaved UV plane) | Source patch required to wire `V4L2_PIX_FMT_NV12` |
| `realsense2_camera` | **No** | Publishes RGB/BGR via librealsense post-processing | NV12 not in librealsense's typical output path |
| `zed_ros2_wrapper` | **No** | RGBA / BGRA via Stereolabs SDK; `enable_24bit_output` for lower bandwidth | No NV12 path in Stereolabs SDK |
| `cv_camera` | **No** | Depends on `cv_bridge` for conversion | cv_bridge has no NV12 support (see below) |

### Cross-cutting blocker — cv_bridge

Direct source check on `ros-perception/vision_opencv` Humble
branch: `cv_bridge.cpp` contains zero references to NV12
(case-insensitive grep). Conversion paths cover MONO8, BGR8,
RGB8, BGRA8, RGBA8, YUV422 variants, Bayer patterns. NV12 is
absent.

This means: even when a driver does emit NV12 (TI fork
of gscam2, today), the standard ROS conversion path
(`cv_bridge::toCvCopy(msg, "bgr8")`) fails. End users today
must hand-roll the NV12 → BGR conversion in their own node.

NV12 was added to `sensor_msgs/image_encodings` in PR #172
(ros2/common_interfaces, 2022) but the matching cv_bridge
conversion never landed.

### One concrete user surface

The TI Robotics SDK fork of gscam2 emits NV12 by default to
feed `ti_vision_cnn` on TI Jacinto SoCs (J7, AM68A, AM62A).
That is exactly the segment Prism positions for —
non-NVIDIA embedded, Isaac ROS does not cover. But it is one
fork on one SoC family, and the v0.1.0 capture set does not
include any TI Jacinto host. Validating F1 against this
real producer requires acquiring a TI host first.

## Implication for F1

User surface on stock Humble today is **effectively zero**:

- No mainline driver emits NV12.
- The one Humble fork that does (TI gscam2) is SoC-specific and
  outside Prism's current capture set.
- The cv_bridge conversion gap means even users who hand-roll a
  NV12 source today cannot consume it through the standard path.

The most leveraged Prism contribution to the NV12 story is
arguably **upstream**: a `cv_bridge` patch to add NV12 → BGR
conversion via OpenCV's `cvtColorTwoPlane`. That unlocks every
NV12-emitting driver future (TI gscam2 today, plus any new
GStreamer-routed driver), not just our wrapper. It is also a
single, reviewable PR to `ros-perception/vision_opencv` rather
than a new Prism node.

A `prism::FormatBridgeNode` would still have value as a
GStreamer-side ingest path, but its win over a hypothetical
patched `cv_bridge` shrinks to:

- `gst_buffer_new_wrapped_full` zero-copy ingest vs cv_bridge's
  copy (a v0.1.0-mechanism DDS-elimination floor)
- direct routing of NV12 inside Prism's GStreamer fragment
  rather than NV12 → BGR → GStreamer

Both are real but smaller than the original F1 hypothesis assumed.

## Recommendation

**Defer F1 for v0.2.** Re-evaluate when one of these changes:

- A TI Jacinto J7 / AM68A / AM62A enters the Prism capture set
  (would give F1 a real user immediately).
- A mainline Humble camera driver lands NV12 emission (would
  expand the user surface meaningfully).
- We ship a separate `cv_bridge` upstream PR for NV12 conversion
  (would unblock the broader ecosystem and let us measure F1's
  delta against a properly-supported baseline).

E1.2 (detour-cost) and E1.3 (prototype) are not run for v0.2 —
without a user surface to deploy against, the timing numbers
would be informational only and the prototype work is better
scheduled when one of the three triggers lands.

## What this finding does NOT change

The architectural insight that NV12 is the lingua franca of every
modern hardware accelerator path is correct, and the v0.1.0
Round-1/2 BGR-CAPS gap on Jetson legacy is a direct consequence of
the same gap. When a triggering event materializes, F1 reactivates.
