We present a runtime-adaptive ROS 2 image-processing node that auto-selects between GStreamer hardware pipelines (NVIDIA NVMM, Intel VA-API) and a direct CPU fallback — preserving the exact parameter interface of image_proc::ResizeNode. The node validates each accelerator against the live GStreamer registry at startup, silently skipping broken ones such as vaapipostproc on GStreamer 1.20. Chainable resize, crop, flip, and colorconvert actions compose inside a single process with scaled CameraInfo tracking the image through each stage — while remaining a two-line change in any existing launch file. Fresh A/B numbers against the current chainable architecture are pending re-measurement.

§1

Motivation.

Hardware accelerators for image operations are abundant on modern robotics hosts — every Jetson has NVMM, every recent Intel SoC has VA-API, and most deploy targets ship with some working GStreamer pipeline. The problem is not availability; it is the cost of using them. A node that depends on a specific backend ties the node to a specific platform, and a node that tries to detect backends at build time loses to the first host that doesn't match the build assumption.

prism::ResizeNode takes the opposite approach: it probes the host at startup, validates against the live GStreamer registry, and lets the fastest actually-working backend win. When nothing works, it still beats image_proc by eliminating the image_transport round-trip through a direct intra-process callback.

1.1 Hardware auto-detection

Startup probes /dev/nvhost-*, /dev/nvmap, and /dev/dri/renderD* to identify present accelerators. A second pass calls gst_element_factory_find to confirm the relevant GStreamer elements are registered and instantiable. Broken accelerators — such as vaapipostproc on GStreamer 1.20 with its chroma-subsampling regression — are skipped, not attempted.

1.2 Hot path: zero-copy ingest, single-copy egress

The subscriber receives a unique_ptr<sensor_msgs::msg::Image> via intra-process transport. A custom appsrc wrapper takes ownership of the underlying buffer via gst_buffer_new_wrapped_full with a GDestroyNotify hooked into the ROS message destructor, moving the pointer — not the pixels — into the GStreamer pipeline. On the way out, the pipeline emits a fresh BGR buffer which is copied into the outgoing sensor_msgs/Image payload: a single copy at egress, and no DDS round-trip.

1.3 Dual-mode fallback

On hosts without a working GPU path, the node bypasses GStreamer entirely and performs the resize with cv::resize in the subscriber callback itself. This path still benefits from zero-copy intra-process delivery and sidesteps all GStreamer initialization overhead.

§2

Positioning.

prism_image_proc is not a NITROS replacement. It targets the segment of the ROS 2 fleet that Isaac ROS does not cover: Intel iGPU, AMD, older Jetson, Jetson Orin on Humble, and non-NVIDIA embedded hosts. The choice is to trade GPU-resident data flow between nodes for portability across hardware that the vendor-supported path excludes.

2.1 The ROS 2 acceleration landscape

Isaac ROS 4.x (current release-4.3) supports only Jetson Thor (T5000/T4000, Blackwell), x86_64 Ampere-or-newer, and DGX Spark — on ROS 2 Jazzy. Isaac ROS 3.2 is the last branch supporting Jetson Orin (Nano/NX/AGX) and Xavier on ROS 2 Humble. Isaac ROS supports no non-NVIDIA hardware.

REP-2007 (type adaptation) and REP-2009 (type negotiation) are the standard ROS 2 zero-copy mechanism since Humble; NITROS is NVIDIA's implementation. prism_image_proc does not implement type adaptation. The design decision is deliberate: a resize node that composes into any existing ROS 2 launch file on Intel, AMD, or Orin-on-Humble is worth more than a GPU-resident one that only runs where Isaac ROS already ships.

ros-gst-bridge (github.com/BrettRD/ros-gst-bridge) provides rosimagesrc/rosimagesink GStreamer elements and is adjacent prior art. prism_image_proc differs by being a ROS 2 component node with a single parameter surface matching image_proc::ResizeNode, live GStreamer-registry validation of detected accelerators, and a dual-mode fallback to cv::resize when no pipeline validates.

2.2 A principled hardware-stack finding

Hardware acceleration on Intel requires GStreamer 1.22+, i.e. Ubuntu 24.04 / Jazzy. On stock Humble (GStreamer 1.20) the vaapipostproc element is present but fails live validation due to a chroma-subsampling regression, and the Intel iGPU path gracefully falls back to direct mode. In that configuration the win comes from eliminating the image_transport + CameraSubscriber DDS round-trip, not from a GPU kernel. On Jazzy hosts or Jetson hosts, the GPU path runs and the kernel offload compounds the win.

2.3 What this node is not

Not a type-adaptation framework — data crosses the egress boundary as a normal sensor_msgs/Image and publishing is a single copy. Not a Jetson Thor or Blackwell target (use Isaac ROS 4.x). Not a replacement for CUDA-resident compute graphs. It is a portable resize node that wins on the hosts Isaac ROS does not serve.

§3

Architecture.

The runtime graph resolves in three stages — probe, validate, construct — and the constructed pipeline differs based on what survives validation. The fallback chain is strictly priority-ordered: a failing probe at tier N drops to tier N+1, never the other way.

┌────────────────┐ startup ──────────▶ │ HardwareDetector │ probe /dev/ │ NVIDIA_JETSON │ │ INTEL_VAAPI │ │ CPU_FALLBACK │ └────────┬────────┘ │ ┌────────▼────────┐ │ validate_platform│ gst_element_factory_find() └────────┬────────┘ │ ┌─────────────────┴─────────────────┐ │ │ ┌────────▼─────────┐ ┌─────────▼────────┐ │ GPU available │ │ CPU fallback │ │ Jetson / vapostproc│ │ direct mode │ │ appsrc → GPU → │ │ cv::resize in │ │ appsink │ │ subscriber cb │ └───────────────────┘ └──────────────────┘
§4

Interface.

The node exposes the exact parameter set of image_proc::ResizeNode. Migration from an existing launch file is a two-token edit: the package string and the plugin class name. No upstream camera driver changes, no image_transport plugin changes, no message conversions.

Unified diff · launch file composable node entry launch/A_B_comparison.launch.py 
1   ComposableNode(2-      package='image_proc',3-      plugin='image_proc::ResizeNode',2+      package='prism_image_proc',3+      plugin='prism::ResizeNode',4       name='resize',5       parameters=[{6           'use_scale': False,7           'height':    480,8           'width':     640,9       }],10   )
Two strings change. The parameters block is parsed by an identical schema on both sides.

The philosophical move is conservative: don't invent a new interface; honour the one the downstream launch system already speaks, and make the faster implementation indistinguishable at the config-file level.

§6

Platforms.

The fallback chain is evaluated in priority order on every startup. Detection is cheap; validation is the expensive step. Tiers that pass probe but fail validation are logged and demoted.

NVIDIA Jetson
TX2, Xavier, Orin. Detected via /dev/nvhost-* and /dev/nvmap. Resize runs on CUDA + NVMM; frames never touch host memory.
nvvideoconvert
Intel VA-API
Gen9+ iGPU or Arc dGPU, GStreamer 1.22+ required (1.20 vaapipostproc fails validation and is demoted). Resize runs on the media engine via GL bridge.
vapostproc
CPU fallback
Any x86_64 or aarch64. No GStreamer initialization. Direct cv::resize in the subscriber callback — still faster than image_proc due to zero-copy intra-process delivery.
cv::resize · direct
§7

Setup.

Ubuntu 22.04 with ROS 2 Humble. For VA-API hosts, ensure gstreamer1.0-vaapi is installed and — if GStreamer is older than 1.22 — expect the direct-mode fallback.

Terminal · ~/ros2_ws bash 
# Clone into a colcon workspace
cd ~/ros2_ws/src
git clone https://github.com/sohams25/prism-ros.git prism_image_proc

# Install dependencies
sudo apt install ros-humble-image-proc gstreamer1.0-vaapi \
  libgstreamer1.0-dev libgstreamer-plugins-base1.0-dev
pip install flask psutil

# Build
cd ~/ros2_ws && colcon build --packages-up-to prism_image_proc
source install/setup.bash

# Launch A/B stress test with any 4K mp4
ros2 launch prism_image_proc A_B_comparison.launch.py \
  video_path:=/path/to/4k.mp4

# In a second terminal — dashboard at http://localhost:8080
ros2 run prism_image_proc visualize_demo.py

The launch file purges stale Fast-DDS /dev/shm locks, terminates orphaned containers from prior runs, and pins a UDPv4-only Fast-DDS profile to silence RTPS_TRANSPORT_SHM initialization noise.
