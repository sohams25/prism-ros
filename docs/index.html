01 — What it is

A hardware-agnostic perception accelerator

Prism is a hardware-agnostic ROS 2 image-processing accelerator. It's a drop-in replacement for image_proc::ResizeNode's resize pipeline — same parameters, same output, with a scaled CameraInfo on the paired topic — that auto-detects host accelerators at runtime and chooses the fastest available backend: a GStreamer pipeline with zero-copy intra-process ingest on supported GPUs (Intel VA-API, NVIDIA Jetson NVMM), or a direct cv::resize fallback when no usable GPU path is present.

  • Drop-in replacement for image_proc::ResizeNode's resize pipeline
  • Auto-detects VA-API / NVMM / CPU at runtime
  • Zero-copy intra-process ingest, single-copy egress, no DDS round-trip
launch.py — one-line swap 
ComposableNode(
    package='prism_image_proc',          # was: 'image_proc'
    plugin='prism::ResizeNode',          # was: 'image_proc::ResizeNode'
    name='resize',
    parameters=[{'width': 640, 'height': 480}],
)
02 — Quick start

Get running in three steps

1

Install prerequisites

On Ubuntu 22.04 with ROS 2 Humble. Pull the image-transport plumbing and the GStreamer dev headers.

shell 
# Ubuntu 22.04 · ROS 2 Humble
sudo apt install \
    ros-humble-image-transport \
    ros-humble-image-transport-plugins \
    ros-humble-image-proc \
    libgstreamer1.0-dev \
    libgstreamer-plugins-base1.0-dev \
    gstreamer1.0-plugins-good \
    gstreamer1.0-vaapi
2

Clone and build

Drop the repo into a colcon workspace and build only the Prism packages.

shell 
mkdir -p ~/prism_ws/src && cd ~/prism_ws/src
git clone https://github.com/sohams25/prism-ros.git
cd ~/prism_ws
source /opt/ros/humble/setup.bash
colcon build --packages-select prism_image_proc
source install/setup.bash
3

Run the demo

Launches a synthetic source feeding a Prism resize node in the same composable container — intra-process, zero-copy where the host supports it.

shell 
ros2 launch prism_image_proc prism_image_proc_demo.launch.py
Prefer the A/B stress test? See the benchmarking section below.
03 — Architecture

Detect, validate, dispatch

At startup, a HardwareDetector probes the host for usable acceleration paths. A PipelineFactory asks gst-inspect to live-validate each candidate — a GStreamer element being present on the system is not the same as it actually working on this kernel, driver stack, and chroma-subsampling combo. The first backend that validates wins.

Fallback chain

PriorityPlatformDetectionProcessing
1 NVIDIA Jetson /etc/nv_tegra_release + nvvideoconvert available GStreamer NVMM zero-copy
2 Intel iGPU / dGPU vapostproc validates via gst-inspect GStreamer VA-API
3 Any x86 / ARM Always available Direct cv::resize

06 — Positioning

Where Prism sits in the ecosystem

Isaac ROS 4.x supports only Jetson Thor, Ampere+ discrete GPUs, and DGX Spark on ROS 2 Jazzy. There is no Humble support and no non-NVIDIA hardware support.

Isaac ROS 3.2 is the last branch that supports Jetson Orin on Humble. Even there, Isaac ROS supports no non-NVIDIA hardware at all.

REP-2007 (type adaptation) and REP-2009 (type negotiation) are the standard ROS 2 zero-copy mechanism, available since Humble. NITROS is NVIDIA's implementation of that contract. Prism does not implement type adaptation — that's a deliberate tradeoff for hardware portability.

ros-gst-bridge is adjacent prior art. Prism differs by being a ROS 2 component node with live-validated backend selection, not a generic bridge.

Principled hardware-stack finding

Hardware acceleration on Intel requires GStreamer 1.22+ (Ubuntu 24.04 / Jazzy). On stock Humble (GStreamer 1.20) the vaapipostproc element is present but fails live validation due to a chroma-subsampling regression; the Intel iGPU path gracefully falls back to direct mode. The upcoming benchmarks will quantify this.

What Prism is not

  • Not a type-adaptation framework. REP-2007/2009 and NITROS already occupy that space.
  • Not a rewrite of image_proc. It's a drop-in resize-pipeline replacement with a wider chain.
  • Not a GStreamer-to-ROS bridge. It's a ROS 2 component that happens to dispatch to GStreamer when useful.
  • Not NVIDIA-only. Intel VA-API and CPU fallbacks are first-class paths.
