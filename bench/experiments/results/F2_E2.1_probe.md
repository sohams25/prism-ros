# F2 — E2.1 capability probe results

Date: 2026-04-30. Branch: `exp/v0.2-validation`.

## Question

Which GPU rectification backends actually validate on each target
host, with the BGR8 ingress/egress that Prism's external API
already pins?

## Intel desktop (BareMetal) — GStreamer 1.20.3, Mesa 23.2.1

| Backend         | Element             | Result   | Notes |
| ---             | ---                 | ---      | --- |
| Mesa GLES       | `glupload ! glshader ! gldownload` | **PASS-but-unviable** | Pipeline reaches PLAYING; processes 4K BGR8 cleanly. Throughput collapse: 14.33 fps / 69.79 ms per frame at passthrough, vs 176.76 fps / 5.66 ms for CPU `videoconvert` passthrough. **12× slower than CPU before any shader work runs.** The cost is BGR↔RGBA upload + GL memory transfer at 4K (24 MB → 32 MB → 24 MB per frame). |
| VAAPI mesh      | `vaapipostproc`     | **FAIL — capability absent** | Element advertises only crop-{top,bottom,left,right} and deinterlace. No mesh, no dewarp, no warp, no LUT property. The "vaapipostproc mesh" backend the v0.2 plan anticipated does not exist on stock Ubuntu 22.04 + Mesa Intel iHD on this host. |

### Method

```
# glshader passthrough probe at 4K BGR
gst-launch-1.0 videotestsrc num-buffers=300 \
  ! "video/x-raw,format=BGR,width=3840,height=2160" \
  ! videoconvert ! glupload ! glshader ! gldownload \
  ! videoconvert ! "video/x-raw,format=BGR" \
  ! fakesink sync=false
# wall: 20.94 s, fps: 14.33, per-frame: 69.79 ms

# CPU videoconvert baseline at 4K BGR
gst-launch-1.0 videotestsrc num-buffers=300 \
  ! "video/x-raw,format=BGR,width=3840,height=2160" \
  ! videoconvert ! fakesink sync=false
# wall: 1.70 s, fps: 176.76, per-frame: 5.66 ms

# vaapipostproc capability inspection
gst-inspect-1.0 vaapipostproc | grep -iE "(dewarp|mesh|warp)"
# (no matches)
```

### Why Mesa GLES fails for Prism's contract

Prism's external API is `sensor_msgs/Image` with `encoding="bgr8"`.
`glshader` operates on `video/x-raw(memory:GLMemory)` with format
`RGBA`. The required upload/download stages dominate per-frame
cost; even an identity shader is 12× the CPU baseline. Adding a
real remap shader on top of this overhead cannot recover.

This is a different kind of failure from the v0.1.0 `vaapipostproc`
chroma-subsampling regression: the regression is a bug, glshader's
overhead is fundamental to BGR ingress/egress with Mesa GL on
this hardware class.

### Why VAAPI fails

The mesh / dewarp / warp capability the experiment plan
anticipated does not exist in the `vaapipostproc` shipped with
GStreamer 1.20.3 on Ubuntu 22.04. It is present in some
Intel iHD-driver builds for newer GStreamer (1.24+) and in
specific OneVPL pipelines, but not on the v0.1.0 anchor host.

## Implication for F2

The hardware rectify backend the v0.2 plan anticipated does not
exist on either v0.1.0 capture host:

| Host          | GPU rectify backend | Result |
| ---           | ---                 | ---    |
| Intel desktop | vaapipostproc mesh  | Capability absent on GStreamer 1.20.3 |
| Intel desktop | Mesa glshader       | Validates but 12× slower than CPU at 4K BGR8 due to upload/download + BGR↔RGBA |
| Orin Nano     | nvvidconv warp      | Capability absent (colorspace + crop + flip only); plus BGR-CAPS gap |

This is not a per-host configuration issue. It is the segment
itself: GStreamer 1.20 + stock distro packaging on the hardware
classes Prism targets does not ship a hardware-accelerated remap
element with a BGR contract. Hosts with newer GStreamer (1.24+)
or vendor SDKs (Isaac ROS / VPI on Jetson, OneVPL on Intel) do
have these capabilities, but they are exactly the segment Prism
explicitly does not target.

## Reframing F2 — CPU/GPU compartmentalization

The operator's directive at this E2.1 sign-off point reframes F2:

> "I would ideally want to use both CPU and GPU in a manner that
> we are getting the best advantage, for eg: we are
> compartmentalising the CPU for deploying lighter operations and
> kind of heavier ones on GPU in such a way that GPU is still
> free for running perception models (which are heavy on GPU
> usage)"

This reframe lands harder than the original "GPU rectify wins
the latency race" hypothesis. The v0.1.0 architectural finding
(GPU adapter tax dominates kernel cost on this segment) plus
this E2.1 finding (no hardware rectify backend exists at all on
this segment) point in the same direction:

- **CPU rectify on Prism's target segment is the right home for
  the operation**, not a fallback.
- **GPU stays free for downstream perception consumers** —
  YOLO, segmentation, depth estimation — that genuinely need
  the parallelism.
- **Isaac ROS / NITROS rectify takes GPU**; Prism rectify does
  not. That is the positioning differentiator, not a latency
  race that Prism cannot win on hardware that doesn't ship the
  capability.

`prism::RectifyNode` for v0.2 therefore deploys as a CPU-only
`cv::remap` wrapper, with the same DDS-elimination floor v0.1.0
already proved measurable. The narrative is "Prism rectifies on
CPU so your GPU stays with the perception stack," not "Prism
rectifies fastest." The latter is a claim Prism cannot
honestly make on its target segment; the former is a claim only
Prism's segment can make.

## Decisions for the rest of the F2 ladder

- **E2.2 — keep, simplified.** Run `cv::remap` at 4K BGR on
  Intel + Orin to establish a per-host CPU absolute timing.
  Useful as a sanity check that 10 Hz 4K rectify is comfortable
  on these CPUs. No GPU comparand.
- **E2.3 — keep, full scope.** A/B `prism::RectifyNode`
  (CPU-only) vs `image_proc::RectifyNode` end-to-end on both
  hosts. This measures the DDS-elimination floor — the actual
  v0.2 win — using the same harness shape as v0.1.0.
- **GPU door deferred, not closed.** When a target host with
  GStreamer 1.24+ enters the v0.2.x roadmap (Intel laptop with
  VAAPI on Ubuntu 24.04, or a non-dustynv Orin image with
  `nvvideoconvert`), revisit GPU rectify as an additive
  backend, not as v0.2's headline mechanism.
