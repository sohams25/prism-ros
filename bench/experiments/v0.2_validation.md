# Prism v0.2 — validation experiments

This document defines the experiments run on `exp/v0.2-validation`
to confirm or falsify two findings from the post-v0.1.0 innovation
scan before any node code lands on `main`:

- **F2 — `prism::RectifyNode`.** GPU/iGPU rectification with CPU
  fallback, addressing image_pipeline issue #1126 (RectifyNode
  performance pain) on the segment Isaac ROS does not cover.
- **F1 — `prism::FormatBridgeNode`.** NV12 ingest as a wire-
  compatible alternative to NITROS's private NV12 type, eliminating
  the forced NV12→BGR CPU detour at ROS ingress.

Each track is a probe → microbenchmark → end-to-end ladder. A
finding is approved for `main` only if the ladder's success
criteria all hold; otherwise it is deferred or dropped with a
written rationale in `bench/experiments/v0.2_findings.md` (created
when the first track closes).

Host scope for v0.2 validation: **Intel desktop** (BareMetal,
GStreamer 1.20, direct-mode fallback) and **Jetson Orin Nano
Super** (dustynv L4T r36.2.0). AMD Ryzen and Qualcomm RubikPi 3
remain on the v0.1.0 roadmap; they are not part of this validation
cycle.

## Settled rules carried from v0.1.0

The following rules from `bench/METHODOLOGY.md` apply unchanged:

- 4K BGR8 source @ 10 Hz from `prism::MediaStreamerNode` for
  every A/B unless the experiment explicitly varies the source
  encoding (E1.2, E1.3 use NV12).
- 120 s capture, 10 s warmup drop, 5 s tail drop.
- Latency is `recv_ns − msg.header.stamp_ns`, captured by the
  same Python rclpy `CaptureNode` on both sides.
- Both A/B sides run inside `component_container` with intra-
  process comms enabled where the upstream side supports it;
  any deliberate asymmetry is documented per experiment.
- No claim is published without a captured run on disk. Drift
  guard: any new median that disagrees with v0.1.0 anchors by
  more than 3 % triggers a HALT.

## Track F2 — `prism::RectifyNode`

### Hypothesis

Prism's `RectifyNode` beats `image_proc::RectifyNode` on every
target host through two distinct sources of win:

1. **DDS round-trip elimination** — present on every host, the
   same architectural source that produces the v0.1.0 resize and
   crop wins on Intel desktop.
2. **GPU shader remap** — present on hosts where
   `glshader` (Mesa GLES), `vaapipostproc` mesh, or `nvvidconv`
   warp validates with a real LUT.

### Falsification

If Prism's *CPU-only* rectify is not faster than
`image_proc::RectifyNode` on at least one target host, F2's
DDS-elimination story is wrong and the finding is dropped.

### E2.1 — Capability probe

**Question.** Which GPU rectification backends actually validate
on each target host with a real fisheye LUT?

**Method.** One `gst-launch-1.0` invocation per host, per
candidate backend, with the LUT pre-baked into the element
properties. Backends to probe:

| Host           | Backend          | Element                         |
| ---            | ---              | ---                             |
| Intel desktop  | Mesa GLES        | `glupload ! glshader ! gldownload` |
| Intel desktop  | VAAPI mesh       | `vaapipostproc` with mesh LUT   |
| Orin Nano      | Tegra warp       | `nvvidconv` with warp LUT       |

Each probe writes a single 4K frame to a file sink and is graded
on whether the pipeline reaches `PLAYING` and produces a non-zero
output. A backend that loads but produces black frames or rejects
the LUT is a fail.

**Success criterion.** Per-host table: backend / pass-or-fail /
notes. Outcome decides whether E2.2 has a GPU comparand on each
host or only the CPU `cv::remap` baseline.

**Deliverable.** `bench/experiments/results/F2_E2.1_probe.md` with
the per-host table and the exact `gst-launch-1.0` lines used.

### E2.2 — Kernel microbenchmark

**Question.** What is the per-frame remap cost at 4K BGR8 on each
host, for each backend that passed E2.1, against a CPU
`cv::remap` baseline?

**Method.** Standalone C++ harness, no ROS. Generate a synthetic
LUT from a representative fisheye + radial-tangential
`camera_info` (parameters fixed in the harness for reproducibility,
documented inline). Run 1000 frames per backend, drop the first
100, report p50 / p95 / p99 in milliseconds.

**Success criterion.** Per-host comparison table: backend / median
ms / p95 ms / fps ceiling. Decides which backends are worth wiring
into the action registry.

**Deliverable.** `bench/experiments/results/F2_E2.2_kernel.md` plus
raw timing CSVs alongside.

### E2.3 — End-to-end A/B

**Question.** Does a CPU-only `prism::RectifyNode` prototype beat
`image_proc::RectifyNode` end-to-end on each target host?

**Method.** Minimal `prism::RectifyNode` prototype implemented on
the `exp/v0.2-validation` branch, using only the existing
`ImageProcNode` direct-mode path with `cv::remap`. No GPU yet.
Add a `rectify` entry to `bench/operations.yaml`. Run via
`bench/run.py` for 120 s on each host with a 4K source. Stock
side: `image_proc::RectifyNode` with the same `camera_info`.

The point of this experiment is to measure the
DDS-elimination floor — the worst case, the win Prism still
gets on every host even when the GPU path collapses.

**Success criterion.** Median Prism latency < median stock latency
on at least one target host. This reproduces the structural shape
of the existing v0.1.0 resize and crop wins.

**Falsification.** If Prism's CPU-only path is slower than
`image_proc::RectifyNode` on both hosts, F2 is dropped.

**Deliverable.** Captures under
`bench/results/v0.2_F2_E2.3_<host>_<timestamp>/` matching the
v0.1.0 capture shape (per-frame CSVs, resource CSVs, meta JSON).
Summary in `bench/experiments/results/F2_E2.3_a_b.md`.

### Decision after E2.3

- E2.1 yields ≥ 1 GPU backend AND E2.3 passes on both hosts:
  deploy F2 with GPU + CPU paths in pipeline_factory.
- E2.1 fails on both hosts AND E2.3 passes: deploy F2 as a
  CPU-only DDS-elimination wrapper, framed honestly.
- E2.3 fails: drop F2; document the negative result.

## Track F1 — `prism::FormatBridgeNode`

### Hypothesis

Eliminating the forced NV12 → BGR CPU detour at ROS ingress
produces a measurable per-frame saving at 4K, and at least one
Humble camera driver in active use today emits NV12 (or can be
forked to do so in ≤ 3 days).

### Falsification

If the detour is < 2 ms at 4K AND no Humble driver in the survey
emits NV12 today AND a fork is > 3 days, F1 is plumbing without
users and is dropped (or deferred until upstream catches up).

### E1.1 — Demand probe

**Question.** Which Humble camera drivers emit `sensor_msgs/Image`
with `encoding == "nv12"` today?

**Method.** Read-only source survey of:

- `ros2_v4l2_camera`
- `gscam2`
- `usb_cam`
- `realsense2_camera`
- `zed_ros2_wrapper`
- `cv_camera`

For each: check the encoding strings the driver publishes, note
whether NV12 is supported (a) out of the box, (b) behind a config
flag, or (c) requires a source patch. Cite line numbers.

**Success criterion.** A per-driver table: NV12 supported / flag /
patch needed / not supported. Outcome decides F1's user surface
size.

**Deliverable.** `bench/experiments/results/F1_E1.1_demand.md`.

### E1.2 — Detour-cost measurement

**Question.** What is the per-frame CPU cost of converting NV12 to
BGR8 at 4K on each target host, on the path Prism would
eliminate?

**Method.** Extend `synthetic_4k_pub_node.cpp` to emit NV12
optionally (new param `output_encoding` defaulting to `bgr8`).
Pipeline: synthetic NV12 → `cv_bridge::toCvCopy(..., "bgr8")` →
`image_proc::ResizeNode` → CaptureNode. The conversion is the
detour we are measuring. Capture for 120 s on each host.

**Success criterion.** Per-host detour median ms. Threshold:

- ≥ 5 ms at 4K → F1 is high-leverage as soon as one driver lands
  NV12.
- 2 – 5 ms → F1 is moderate, contingent on E1.1 demand.
- < 2 ms → F1 is plumbing; drop unless E1.1 surfaces multiple
  immediate users.

**Deliverable.** Captures under
`bench/results/v0.2_F1_E1.2_<host>_<timestamp>/`. Summary in
`bench/experiments/results/F1_E1.2_detour.md`.

### E1.3 — Prototype path

**Question.** End-to-end, what is the achievable delta if Prism
ingests NV12 directly?

**Method.** Minimal `prism::FormatBridgeNode` prototype on the
`exp/v0.2-validation` branch: accept `sensor_msgs/Image` with
`encoding == "nv12"`, build a GStreamer pipeline with
`video/x-raw,format=NV12` ingress caps via the existing
`pipeline_factory.cpp` machinery, emit BGR8. A/B against the
detour path from E1.2.

**Success criterion.** End-to-end median ms on the FormatBridge
path. Achievable delta = E1.2 detour − E1.3 path. Decides F1
go/no-go.

**Deliverable.** Captures under
`bench/results/v0.2_F1_E1.3_<host>_<timestamp>/`. Summary in
`bench/experiments/results/F1_E1.3_prototype.md`.

### Decision after E1.3

- E1.1 yields ≥ 1 wire-compatible driver OR fork ≤ 3 days, AND
  E1.2 detour ≥ 5 ms at 4K, AND E1.3 delta ≥ 3 ms: deploy F1.
- Any criterion fails: defer F1 until the missing piece lands
  upstream or a host with a native NV12 producer enters the
  capture set.

## Order of operations

1. F2 track first. F2 has higher confidence going in and shares
   structural shape with v0.1.0 wins (DDS-elimination floor is
   already a measured phenomenon on this codebase).
2. F1 track second.
3. After both tracks close, consolidate findings in
   `bench/experiments/v0.2_findings.md` and surface the deploy /
   defer / drop decision per finding to the operator. No node
   code merges to `main` until that decision is made.

## What is out of scope for v0.2 validation

- F3 (`/diagnostics` publication). Pure implementation; no
  hypothesis to falsify. Tracked separately.
- F4 (`prism_image_transport` decode plugin). LEAVE-IT-ALONE per
  innovation-scout; revisit when ≥ 2 of the v0.1.0 roadmap host
  captures (RubikPi, RK3588, AMD) exist on disk.
- F5 (sensor-pipeline QoS recipe doc). Documentation pass; not an
  experiment.
- AMD Ryzen and Qualcomm RubikPi 3 captures. v0.1.0 roadmap items;
  not part of this cycle.
- Edits to `~/roscon2026-materials/`. Proposal is submitted.
- Edits to v0.1.0 settled surfaces (CLAUDE.md hard rule 6).

## Branch and commit policy

- All experiment work lives on `exp/v0.2-validation` until a
  finding is approved.
- Per CLAUDE.md hard rules: HALT before push, plain-English
  commits, no claims without measurement, joint-surface
  agreement once any v0.2 numbers reach README or site.
- v0.1.0 anchor numbers in README and site stay frozen; v0.2
  numbers, if any reach those surfaces, land as additional rows
  with explicit version annotation.
